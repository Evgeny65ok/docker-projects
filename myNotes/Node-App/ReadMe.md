# Отчёт: JavaScript в Docker — магия Node.js без головной боли

## 1. Подготовка: пара движений — и готово

Создали скромный, но гордый файл `app.js` — он всего‑то и делает, что выводит приветствие в консоль. Минимум кода, максимум смысла!

## 2. Запуск скрипта: Docker + Node.js = дружба навеки

Запустили JavaScript‑код внутри контейнера на базе легковесного образа `node:alpine`. Хитрость в томах: они «подружили» локальный файл с изолированным окружением.

Команда запуска:

`docker run --rm -v "$(pwd):/app" node:alpine node /app/app.js`

Что здесь важно:
* `--rm` — контейнер удалится сам после выполнения, не оставляя следов;
* `-v "$(pwd):/app"` — монтируем текущую директорию в `/app` внутри контейнера;
* `node:alpine` — берём компактный образ, чтобы всё работало шустро.

## 3. REPL‑приключения: мгновенный JavaScript

Протестировали интерактивный режим REPL (Read‑Eval‑Print Loop) — это как мини‑лаборатория для экспериментов с кодом прямо в окружении Node.js. Можно:
* проверять логику на лету;
* тестировать небольшие фрагменты кода;
* отлаживать выражения без сохранения в файл.

Быстрый старт REPL:

`docker run -it --rm node:alpine node`

## 4. Вывод: почему это круто

Запуск Node.js в Docker — это:
* **Стандартизация:** код работает одинаково на любой машине с Docker — никаких «у меня на ноутбуке всё было хорошо!»
* **Изоляция:** зависимости и версии Node.js не конфликтуют с хостовой системой.
* **Портативность:** собрал образ — и отправил коллегам или в продакшн.
* **Лёгкость:** образ `node:alpine` занимает минимум места, но даёт полноценную среду выполнения.

**Итог:** Docker превращает запуск JavaScript‑скриптов в предсказуемый и удобный процесс — больше никаких сюрпризов с окружением!
